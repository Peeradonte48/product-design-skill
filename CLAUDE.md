# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

This is a **skill-only scaffold**, not an application. It contains twelve portable, stack-agnostic Claude Code skills under [skills/](skills/), plus the `install.sh` installer and a `VERSION` file — no `package.json`, no source tree, and no test setup. There are therefore no build/lint/test commands to document. If real app code is ever added, update this file with the actual commands.

The only other tracked content is [docs/](docs/), which is **ADRs only** (`docs/adr/0001–0003` — architecture decision records referenced from this file, the README, and a couple of skills). `docs/` is gitignored and **not shipped** to users (the installer copies `skills/` only), so new files there need `git add -f`. The one-time brainstorm specs and implementation plans that used to live under `docs/superpowers/` were removed — don't expect them, and don't re-add SDD process artifacts as tracked files.

The skills are deliberately **not** tied to any one project, framework, or design system. They detect and conform to whatever target project they're run against. (An earlier version of this repo hardcoded a "Ramen POS / Next.js / Tailwind" target; that coupling has been removed — do not reintroduce project-specific assumptions into the skills.)

## The skill suite

The twelve skills form a **FigJam → narrative → prototype** pipeline, plus a **fusion** path (finished design + flow → working prototype), three standalone Figma paths (design→code, sitemap→spec, and running-page→Figma), **two command-only read-only Figma checks** (`critique-figma-design`, the single-frame self-check, and `verify-design-match`, the running-page↔Figma parity audit), one **command-only Figma→dev-docs generator** (`figma-to-dev-docs`, which turns a finished frame/section into a PRD + spec-driven spec + Gherkin test-cases bundle), and three **no-Figma doc skills**: two **command-only doc-review** commands (`harden-doc`, `biz-review`) that run only on explicit `/harden-doc` / `/biz-review` invocation (`disable-model-invocation: true`), and one **generative producer**, `spec-to-brief`, that writes a stakeholder brief from an existing spec/UCN. The Figma/FigJam skills read via the Figma MCP plugin and never guess, and every skill opens with a "clarify until clear" rule: stop and ask the user rather than inventing a missing detail.

- **[implement-figma-design](skills/implement-figma-design/SKILL.md)** — *standalone, design → code.* Transcribes a finished Figma frame into a 1:1, pixel-perfect build in the target codebase. It first extracts the frame into a persisted single-owner `design-spec.md` (per `skills/implement-figma-design/references/design-spec-format.md`) and builds from that, then verifies via an inline per-element ✓/Δ summary anchored on the Figma screenshot as the oracle. It emits **no report** — formal parity reports are `verify-design-match`'s alone (see `docs/adr/0004-implement-figma-design-emits-no-report.md`). Use when the user shares a figma.com link and wants it built/matched. This is the web/React path; it does **not** push code into Figma (that's `figma-generate-design`) and does **not** build Flutter/mobile (that's `implement-figma-design-mobile`).

- **[figjam-to-use-case-narrative](skills/figjam-to-use-case-narrative/SKILL.md)** — *pipeline step 1, FigJam → narrative.* Reads a user-flow diagram from FigJam (via `get_figjam`) and writes a structured use-case-narrative (UCN) markdown doc. Reads only; never edits the board.

- **[use-case-narrative-to-prototype](skills/use-case-narrative-to-prototype/SKILL.md)** — *pipeline step 3, narrative → code.* Turns a UCN doc into a walkable, clickable code prototype (behavioral fidelity, not pixel fidelity; React by default, stack-aware). When a finished Figma design exists and pixel fidelity is needed, defer to `implement-figma-design`.

- **[figma-design-to-working-prototype](skills/figma-design-to-working-prototype/SKILL.md)** — *fusion, finished design + flow → working prototype.* A **thin orchestrator** that fuses a finished Figma design (pixels) with its use-case-narrative (behavior) into one prototype that is **both** pixel-perfect **and** walkable. Behavior-first: it delegates the walkable **skeleton** to `use-case-narrative-to-prototype`, then **re-skins** each screen in place (presentation only, wiring untouchable) via `implement-figma-design`, gated by a two-phase check (behavioral walk → per-(screen × state) pixel diff → behavioral walk again). Fires **only when both a finished Figma design and a flow/UCN are present** for the same screens; single-input cases route to the relevant sibling. **This is the second skill with a hard dependency on sibling suite skills** (after `page-to-figma`) — see `docs/adr/0002-p1-composes-sibling-suite-skills.md`. Model-invocable. Needs the Figma MCP read tools transitively via its siblings.

- **[figjam-sitemap-to-spec](skills/figjam-sitemap-to-spec/SKILL.md)** — *standalone, FigJam → spec.* Reads a sitemap / site-structure diagram from FigJam (via `get_figjam`) and writes a product spec markdown doc (sitemap tree + per-page specs). Reads only; never edits the board. Standalone, but the spec is plain markdown that composes with the suite's doc-review commands (`/harden-doc`, `/biz-review`) and the build skills above. Its `references/product-spec-guide.md` is flexible guidance, not a strict template, and is single-owner (not part of the shared contract below).

- **[page-to-figma](skills/page-to-figma/SKILL.md)** — *standalone, running page → Figma.* The inverse of `implement-figma-design`: transcribes a **running** product page into a 1:1 Figma frame. It is an **accuracy orchestrator** on top of the **vendored figma-cli** (local CDP, no rate limit), falling back to the official Figma plugin only when the CLI is unavailable: it extracts live-DOM computed styles as ground truth, delegates bulk assembly to figma-cli's `render`/`render-batch`, and gates on a **deterministic numeric property read-back** — **position-first** (every node's absolute x/y + container alignment vs the DOM rect, not just intrinsic size/padding/color/type) and not a screenshot diff — **plus a structure gate and an extraction-completeness gate** (shadow-DOM/pseudo-element walking, and rasterizing the unwalkable like `<canvas>`/cross-origin `<iframe>` so nothing ships silently dropped) (the build target is a *nested auto-layout tree mirroring the DOM box model*, not a flat pile of absolutely-positioned siblings — flat is a per-subtree, logged last resort), correcting wrong properties with batched `eval` writes until both the numeric checklist and the structure gate are green. It has a first-class **Breakpoint mode** (re-extract + full build per viewport width, sibling to Flow mode) and a **CSS→Figma fidelity map** (`references/css-figma-map.md`) for rich CSS — gradients, blur/glassmorphism, blend modes, `object-fit`, transforms, `text-transform`, mixed inline runs, per-side/dashed borders, and `oklch`/P3→sRGB color normalization. **This is the first suite skill with a hard dependency on another plugin's skills** (`figma-use`, `figma-generate-design`) — see `docs/adr/0001-page-to-figma-depends-on-official-figma-plugin.md`. As of v1.8.0 that dependency is the *fallback*: the primary engine is the vendored figma-cli (see `docs/adr/0006-page-to-figma-vendored-figma-cli.md`). Contrast: `implement-figma-design` is Figma → code; the official `figma-generate-design` is the distrusted code → Figma path this skill supervises.

- **[critique-figma-design](skills/critique-figma-design/SKILL.md)** — *command-only, Figma frame → objective critique.* A **read-only** self-check that runs a finished Figma frame through an objective checklist and returns a severity-ranked report: four **measured** categories (accessibility, design-system consistency, structure/hierarchy, layer hygiene — each finding citing value · threshold · source) plus an evidence-anchored **Nielsen-10 heuristic** pass kept in a separate section. It is a **self-check, not a taste-maker** — never a bare aesthetic preference, and **no quality score**. Contrast is measured by screenshot-pixel-sampling; thresholds conform to the target project's rules first (else WCAG AA + platform defaults); touch-target is modality-aware; role-dependent checks require a cited role-signal. **Command-only** (`/critique-figma-design`, `disable-model-invocation: true`) so it never fires uninvited or collides with `implement-figma-design`. Read-only Figma MCP; depends on no other suite skill. See `references/check-catalog.md`.

- **[verify-design-match](skills/verify-design-match/SKILL.md)** — *command-only, running page + Figma frame/section → multi-state parity audit.* A **read-only** check that compares a **running page** against the finished Figma frame(s) — **or a whole section of state-variant frames** — it should match and reports, **per breakpoint and per state**, where the implementation diverges. It is **read-only on the codebase and Figma** (never edits code, never writes to Figma); to reach server-driven states it may drive the **live app under test**, but **only with explicit user authorization** (each destructive step is named in the report). Detection is a **visual pass** (screenshot overlay locates diverging regions) then a **property pass** (measured live-DOM-vs-Figma diff); element matching is **geometry-first/text-anchored**, and pairs below the confidence bar go to a **"Couldn't align"** list rather than being guessed. Section inputs are handled **auto-then-ask-gaps**: it auto-matches the default state and asks the user for the route + interaction recipe of states it can't reach. Output is a **per-category verdict** (✓/⚠/✗ for color, typography, spacing, layout, sizing) plus severity-ranked, cited findings — **no overall score** — emitted as **both a human PDF and an AI-agent Markdown handoff** (two layers: machine-readable findings + an `implement-figma-design`-ready fix list; the Markdown composes downstream but the skill never invokes the fixer itself). **Command-only** (`/verify-design-match`, `disable-model-invocation: true`) so it never fires uninvited or collides with `implement-figma-design`. It has a **hard, fail-closed tooling dependency** — Playwright (MCP or CLI) **and** Figma access (MCP read tools or REST + token); it stops if either is missing — making it the second suite skill (after `page-to-figma`) with a hard external-tool dependency (PDF/Markdown output needs no new dependency — PDF renders via the same Chromium, Markdown is a file write). See `references/parity-check-catalog.md` and `docs/adr/0003-verify-design-match-requires-playwright-and-figma-access.md`.

- **[harden-doc](skills/harden-doc/SKILL.md)** — *doc-review command, doc → hardened doc.* Interviews the user relentlessly about a plan/spec/UCN — one question at a time, a recommended answer with each — until every branch of the decision tree is resolved. Fork of the standalone `grill-me`, renamed and adapted to suite style; **command-only** (`/harden-doc`, `disable-model-invocation: true`). Resolves **ambiguity**; for premise/scope challenge, route to `biz-review`. Needs no Figma.

- **[biz-review](skills/biz-review/SKILL.md)** — *doc-review command, doc → scope decisions.* Challenges a plan/spec/UCN from the founder lens: premise challenge, demand-evidence forcing questions, mandatory alternatives (minimal-viable vs ideal), then a four-posture scope review where **every scope change is an individual user opt-in**. **Command-only** (`/biz-review`, `disable-model-invocation: true`). Distilled from gstack's `plan-ceo-review` + `office-hours` (MIT, © 2026 Garry Tan) with the gstack runtime, telemetry, and state machinery removed — **keep it portable; do not reintroduce gstack binary calls or `~/.gstack` paths.** Needs no Figma; uses WebSearch for the landscape check when available.

- **[spec-to-brief](skills/spec-to-brief/SKILL.md)** — *generative doc, spec/UCN → product brief.* Synthesizes a stakeholder-ready product brief / PRD-lite from an existing product spec, UCN, sitemap, or described idea — problem, target user, demand evidence, goals & metrics, solution overview, scope, handoff. **Model-invocable** (unlike the two review commands), but it composes *after* them: run `/biz-review` (challenge) and `/harden-doc` (resolve) first, then this writes up what survived. It **synthesizes** and asks only for the business inputs a design artifact can't hold — it never challenges premise (`biz-review`) or interviews to resolve ambiguity (`harden-doc`), and it **never fabricates a metric or demand claim** (unsourced facts go to Open Questions). Distilled in style from gstack's `to-prd` / `spec` *content* with the gstack runtime and issue-tracker publish removed — **keep it portable.** Its `references/product-brief-template.md` is flexible guidance, single-owner (not part of the shared contract below). Needs no Figma; honors the principle that the designer owns the Figma design itself — the brief hands off, it doesn't design.

- **[figma-to-dev-docs](skills/figma-to-dev-docs/SKILL.md)** — *command-only, finished Figma frame/section → AI-developer doc bundle.* A **read-only** skill that reads a finished frame or section (Figma MCP read tools), confirms the screen/state mapping, runs a **batched-by-topic context interview** for the behavior/business a static design can't hold, then writes three files to a confirmed per-feature folder: a **developer-facing `PRD.md`** (build context, *not* a buy-in brief), a spec-driven **`spec.md`** (one file: EARS requirements with `REQ` ids · stack-aware design with **no pixel geometry** · ordered tasks citing `REQ` ids), and a Gherkin **`test-cases.md`** (scenarios tagged by `REQ` id, serving as the acceptance criteria) — keeping a **REQ → task → test** traceability chain. **Command-only** (`/figma-to-dev-docs`, `disable-model-invocation: true`) so it never collides with `implement-figma-design` (which also fires on a figma link, but to *build code*). Its interview is **generate-only** — premise challenge routes to `biz-review`, ambiguity-resolution to `harden-doc`, a buy-in brief to `spec-to-brief` — and it **never fabricates a metric or demand claim**. Read-only Figma; reads the codebase read-only when in a repo to ground the docs; **no hard dependency on any sibling skill**. See `references/doc-templates.md`, `references/context-interview.md`, and `docs/adr/0005-figma-to-dev-docs-separate-from-spec-to-brief.md`.

## Shared format contract — keep in sync

Both `figjam-to-use-case-narrative` and `use-case-narrative-to-prototype` carry an identical copy of `references/use-case-narrative-format.md` (the canonical UCN template — one skill writes it, the other reads it). These two copies are a shared contract: **when you edit one, edit the other to match.** Each skill also has a one-directional mapping reference (`references/figjam-mapping.md` and `references/prototype-mapping.md` respectively).

## Distribution & versioning

The suite is distributed **curl | bash only** via `install.sh` (npm and the Claude Code plugin marketplace were deliberately declined — don't reintroduce them unasked). The repo-root [`VERSION`](VERSION) file is the suite's single version of record: `install.sh` stamps it into a `~/.claude/skills/.product-design-skill.version` manifest, and `--update` / `--check` read it so already-installed users can update easily. **Bump `VERSION` whenever you change shipped content** (anything under `skills/` **or `commands/`**) — otherwise `--check` will tell users they're current when they aren't. Docs-only changes (README/CLAUDE.md/ADRs) don't need a bump. The `install.sh` **`SKILLS` array must list every shipped skill** — a skill missing there is silently never installed (this is how `figma-to-dev-docs` shipped uninstalled until v1.4.0).

Alongside `skills/`, the repo ships a **`commands/`** dir of single-file Claude Code slash commands (e.g. `update-design-skills.md`, which runs `--update` from inside a session). The installer copies these into the **sibling `commands/` dir of the skills target** (`~/.claude/commands` for a user install, `<proj>/.claude/commands` for `--project`) with the same selective-update / prune-on-removal / `--uninstall` handling skills get, and the manifest carries a `commands=` line for pruning. The `install.sh` **`COMMANDS` array must list every shipped command filename** — same gotcha as `SKILLS`: a command missing there is silently never installed.

User-facing changes are logged in the repo-root [`CHANGELOG.md`](CHANGELOG.md) (Keep a Changelog style) — **add an entry on every `VERSION` bump**; the update notice and installer output link to it.

**Vendored figma-cli.** The repo ships a vendored copy of `silships/figma-cli` (MIT) under
repo-root `figma-cli/`, pinned to upstream tag v2.1.0 (see `figma-cli/VENDORED.md`). It is the
**primary Figma engine for `page-to-figma`** (local CDP — no API token, no rate limit), with
the official Figma plugin MCP retained as an announced fallback. `install.sh` copies
`figma-cli/` to the sibling `figma-cli/` dir of the skills target (`~/.claude/figma-cli`) and
runs `npm install` best-effort (non-fatal if Node is absent); the manifest carries a
`figma-cli=<version>` line and `--uninstall` removes it. Vendored files are a clean upstream
copy — **never hand-edit them**; update via a deliberate re-vendor. Note: the npm package
`figma-use` (a CLI dependency) is *not* the official Figma plugin skill `figma-use`.

**Update notifications.** As of v1.4.0 the installer writes a Claude Code `SessionStart` hook (into `settings.json`, via `python3` or `jq`) that checks GitHub's raw `VERSION` at most once per 24h and prints a one-line "update available" notice — on by default, `--no-notify` opts out, `--uninstall` removes it, fails silent offline. The notifier script is generated inline by `install.sh` (a `%q`-injected header + a quoted heredoc body), not stored as a separate repo file, so the raw `curl | bash` one-liner stays self-contained. Keep it **fail-closed and non-blocking** — a notifier must never delay or break a session.

## Conventions when editing these skills

- **Stay stack-agnostic.** Skills detect the target project's framework, file structure, component patterns, styling, and token system and conform to them. Don't bake in a specific stack, design system, or domain.
- **Honor the target project's own conventions** (token systems, accessibility/ergonomics rules like touch-target or body-font minimums) while matching the design — and surface a conflict rather than silently shipping something that breaks a rule.
- **Reuse over reinvention; match every breakpoint.** Use existing components/tokens; a match at one width that breaks at another is not a match.
- **Clarify until clear.** This is the cross-cutting rule in all the skills — a wrong guess that looks right silently corrupts the spec. Batch related questions, but don't proceed past an unresolved ambiguity.
- **Don't confuse direction.** Most of these skills go design/flow → code (or doc); `page-to-figma` is the one exception, going running-page → Figma. Three directions stay distinct: pushing a *finished Figma design into code* is `implement-figma-design`; pushing *raw code into Figma* with no accuracy supervision is the separate `figma-generate-design` / `figma-generate-library` suite; `page-to-figma` is the supervised running-page → Figma path that orchestrates that suite. Don't reach for one when the user wants another. A fifth review/QA-type skill, `verify-design-match`, is **command-only** and **read-only**: it compares a *running page* against its *Figma frame* (live ↔ design parity) and never edits either side — distinct from `implement-figma-design` (which builds/fixes code), `page-to-figma` (which writes Figma), and `critique-figma-design` (which checks one frame against rules). Keep its `disable-model-invocation: true` and its read-only posture.
- **Keep the fusion skill thin and behavior-first.** `figma-design-to-working-prototype` must stay a thin orchestrator of `use-case-narrative-to-prototype` (skeleton) + `implement-figma-design` (re-skin), in that order — don't reverse to pixels-first and don't inline either sibling's logic. Its re-skin contract is **presentation only, wiring untouchable**; preserve it. See `docs/adr/0002-p1-composes-sibling-suite-skills.md`.
- **Load `figma-use` before any `use_figma` write call.** Pure reads (`get_design_context`, `get_screenshot`, `get_metadata`, `get_variable_defs`, `get_figjam`) generally don't need it.
- **Keep the three doc lenses distinct.** Three skills operate on docs without Figma: `harden-doc` *resolves* ambiguity, `biz-review` *challenges* premise/value/scope, and `spec-to-brief` *produces* the stakeholder brief from what survives. The two review skills are **command-only** — don't remove their `disable-model-invocation: true` (they must never auto-trigger); `spec-to-brief` is model-invocable because it's a generative producer, not a review. Their "When NOT to use" sections route to each other — keep that routing intact when editing any of them, and keep `spec-to-brief` from drifting into challenging or interviewing (it synthesizes and never fabricates a metric). A fourth review-type skill, `critique-figma-design`, is also **command-only** but is a *read-only Figma self-check* (objective + heuristic), not a doc lens — keep its `disable-model-invocation: true` too, and keep it from drifting into aesthetic taste-making (it cites measured values or named heuristics, never bare preference). A fifth command-only skill, `figma-to-dev-docs`, is also a *generative producer* but **command-only** (unlike `spec-to-brief`) so it never collides with `implement-figma-design` on a figma link — keep its `disable-model-invocation: true`, its read-only posture, and its developer-facing-PRD vs `spec-to-brief` buy-in-brief boundary (ADR 0005).
- **Compose with external plugins, don't rebuild.** Some inputs are better borrowed than built. For **landscape / competitive evidence**, `biz-review` and `spec-to-brief` should lean on the `deep-research` plugin (or `WebSearch` offline) and feed in its *cited* output rather than inventing a market claim. For **user-research synthesis** (interview notes → insights), route to an existing plugin like `interview-insights` rather than writing a suite-native version. Keep these as **pointers, not hard dependencies** — the suite skills stay runnable without them (unsourced claims still go to Open Questions; the designer can still supply evidence). Don't reintroduce a suite-native research/synthesis skill that duplicates these.

## Tooling available

Figma MCP read tools (`get_design_context`, `get_screenshot`, `get_metadata`, `get_variable_defs`, `get_libraries`, `search_design_system`, `get_figjam`) and write tools (`use_figma`, plus `generate_figma_design` — the running-app capture `page-to-figma` uses to source image hashes), the `shadcn` MCP, and browser/screenshot tooling (e.g. Playwright) for DOM extraction and screenshot-based verification.
